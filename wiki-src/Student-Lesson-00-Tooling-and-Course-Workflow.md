# Student Lesson 00: Tooling and Course Workflow

This lesson is about learning how to participate in the course without fighting the tools first.

## Main Ideas

Programming classes often feel harder than they really are because students are trying to learn ideas and tools at the same time. In this course, we want the tools to become boring as quickly as possible. If you can open the project, switch to the correct lesson branch, build the code, run tests, and start simulation, then the rest of your attention can go toward learning Java and FRC.

A repository is the full project. It includes source files, build files, Git history, and supporting folders. A branch is one saved version of that repository. In this course, each lesson has its own branch so students can move to a clean snapshot of the code for that lesson.

The VS Code Source Control view and Git Graph are the Git tools for this course. Students do not need to memorize command-line Git to succeed here. What matters first is recognizing whether the project is clean, whether files were changed, and whether it is safe to move to a different lesson branch.

The WPILib command palette is the main way students run the project. `Build Robot Code` checks whether the code can be turned into a program. `Test Robot Code` runs automatic checks of behavior. `Simulate Robot Code` runs the code on the computer as if the computer were standing in for the robot.

## What Students Should Understand

- The project is not one file. It is a folder structure with history and tools around it.
- A branch is not a separate project. It is a different snapshot of the same project.
- The first goal is reliable workflow, not speed.
- If the tools are confusing, the right response is to slow down and identify which step is failing.

## Good Habits

- Check which branch you are on before editing.
- Keep Source Control clean when you move between lessons.
- Build or test early instead of waiting until many changes pile up.
- Treat tool errors as information, not as proof that you are bad at programming.
