# Lesson 01 Reading: Reading a Java Project

## Big Picture

When beginners open a programming project, they often look for a single "main file" and assume everything important must happen there. Real projects usually do not work that way. A Java project is a collection of files and folders with a structure. That structure is not random. It exists to help programmers know where source code lives, where tests live, where build tools keep their settings, and which files are part of the project history instead of the program itself.

Learning to read a project starts before reading code line by line. Students should first ask broader questions. What kind of project is this? Where is the actual source code? Where are the tests? Which files exist for Git? Which files are generated by tools? Once those questions are clear, the code becomes much easier to navigate. This is one reason experienced programmers can enter a new codebase more quickly than beginners: they read the structure before they read every detail.

In this course, students are reading an FRC Java project. That means the project combines normal Java structure with files added by WPILib, Gradle, and Git. Students do not need to master every robot-specific framework file right away. In the early Java block of the course, the goal is simpler: know what belongs to the program, what belongs to testing, and what belongs to the development tools. The deeper FRC structure is taught later.

## Key Terms

- directory structure: the planned layout of folders and files in a project
- package: a named group of related Java classes
- package statement: the line near the top of a Java file that says which package the class belongs to
- import: a way for one Java file to use a class from another package
- JAR file: a packaged Java output file that can hold compiled classes and other resources
- source code: the files programmers write to define program behavior
- generated file: a file created by a tool rather than written by hand
- entry point: the place where program execution begins

## How Java Projects Are Usually Organized

In Java, classes are commonly grouped into packages. A package name tells you where a class fits in the larger project. For example, classes that work together in one area are often placed in the same package. That helps students answer questions like "Which files belong together?" and "Which code is probably related?" In many Java projects, one class lives in one file, and the file name matches the class name. That pattern makes it easier to find code quickly.

Students should also learn that Java packages usually follow a naming convention. In many Java libraries and business projects, the package name starts with a reversed internet domain, such as `com.example.project`, `org.teamname.robot`, or `edu.wpi.first.wpilibj`. The reason is simple: domain names are usually unique, so reversing them gives projects a naming system that is less likely to collide with someone else's code. Students do not need to memorize domain ownership rules in detail, but they should understand the pattern when they see it.

In FRC, students will often see shorter package names too. The default WPILib project template commonly uses `frc.robot`, which is simpler than a full reverse-domain package name but still follows the same general idea of grouping related code under a package path. A team could choose something longer, such as `com.kauaibots.frc.robot`, but many FRC examples stay with `frc.robot` because it is easy to read and matches the template structure students see early on.

Imports connect those packages. If one file needs to use a class from another package, it usually imports that class. Students should think of imports as part of a file's vocabulary. The import list tells you which outside ideas this file depends on. If a file imports many testing libraries, it is probably a test file. If it imports many build or framework classes, it may be one of the higher-level project files that students will study in more detail later. Imports are often a very fast way to guess a file's purpose before reading the body.

Students should also understand that the project tree is a communication tool. A folder name like `src/main/java` tells you, "This is real program code." A folder like `src/test/java` says, "This is test code." A folder like `build` usually contains generated output. Once students learn these patterns, they stop treating every file as equally important and start reading with better focus.

## Packages and Folders

One important Java rule is that package structure and directory structure are closely connected. If a file says `package frc.robot.util;`, Java expects that file to live in folders that match that package path under the source root. In other words, the file would normally be inside `src/main/java/frc/robot/util/`. Each word in the package name becomes another folder level.

This is why package names are not just labels. They help determine where files belong in the project tree. Students should get used to reading the package statement near the top of a file and connecting it to the folders around that file. This makes navigation much easier. If a class is in package `frc.robot.util`, the student already has a strong guess about where that file lives.

## Important Folders and Files in This Course

The `.git` folder stores the repository history and internal Git data. Students normally do not open or edit it directly. It exists so Git can remember commits, branches, and file history. The `.gitignore` file tells Git which files should usually stay out of version control. In this course, that includes tool-generated files that would only create noise in Source Control. Students do not need to memorize the full `.gitignore` syntax on day one, but they should know its purpose: keep accidental clutter out of the repository.

The `src` folder contains source code. Inside it, `src/main/java` holds the program itself, and `src/test/java` holds tests. This separation matters because tests are part of the project, but they are not deployed to run the robot in the same way the main code is. Students should get used to asking, "Am I looking at program code or test code?" before they try to understand a file.

The `gradle` folder and files like `build.gradle` or `settings.gradle` belong to the build system. They help define how the project is compiled, tested, and assembled. Students do not need deep Gradle knowledge yet, but they should recognize that these files control project setup rather than robot behavior. The `vendordeps` folder stores JSON files that describe outside robotics libraries. In FRC, vendor libraries are common because teams often use hardware that comes with its own software package.

The `build` folder usually contains generated output. Students should not treat it like source code. It is produced by the tools and can often be recreated. That difference between source files and generated files is important. If a student edits a generated file, they may be changing something that will be overwritten the next time the project builds.

One kind of generated output students may see there is a `.jar` file. `JAR` stands for `Java ARchive`. A JAR file is a package file that can hold compiled Java classes and other resources such as configuration files. Students can think of it as a bundled Java output. They do not usually edit a JAR directly. It is something the build system creates from the source code and resources.

## Naming Conventions and Case Styles

Java uses naming conventions to help readers guess what something is before they know every detail. `camelCase` usually names variables and methods. It starts with a lowercase letter, and each later word begins with an uppercase letter, like `targetSpeed` or `calculateDistance`. `PascalCase` usually names classes, such as `RobotConfig` or `ShooterPreset`. `UPPER_SNAKE_CASE` is commonly used for constants that should not change, such as `MAX_SPEED`. `snake_case` is not a common style for normal Java code, but students may still see it in filenames, JSON keys, or other tools outside normal class naming.

These styles are not just decoration. They help a reader identify roles quickly. If students see `ShooterSubsystem`, they should suspect it is a class. If they see `driveSpeed`, they should suspect it is a variable or method. If they see `DEFAULT_TIMEOUT`, they should suspect it is a constant. Naming patterns reduce confusion because they give extra clues before the reader even understands the logic.

## How To Read a Project Without Getting Lost

A good first pass through a new project is broad, not deep. Look at the top-level folders. Find `src/main/java`. Identify the package structure. Look for the entry point file. In this course, that often means finding `Main.java`, `Robot.java`, and nearby project files without trying to master all of their framework roles yet. Then scan test files and build files. This top-down approach helps students build a map before they worry about individual methods.

Students should not panic when they see unfamiliar files. A project is allowed to contain things the student does not fully understand yet. The goal of this lesson is not complete mastery. The goal is to know how to categorize what you are seeing. Is this a source file, a test, a build file, a Git file, a generated file, or a library description? Once that category is known, the file becomes less mysterious.

## Why This Matters For FRC

FRC projects are usually larger than the first toy programs students see in a beginner Java class. That can feel intimidating at first. The directory structure is what makes that size manageable. If students know where program code lives, where tests live, and which files belong to the toolchain, they can move through the project with much more confidence even before they learn the deeper robot framework.

Build season also moves quickly. Students often need to answer practical questions about where code lives and which files matter first. Those answers come faster when the student already understands the project shape. Reading project structure is therefore a real robotics skill, not just an academic exercise.

## Before You Move On

- You should know what a directory structure, package, and import are.
- You should understand that package names often follow a reverse-domain naming style and that package names usually match folder structure under the source root.
- You should know that a JAR file is a Java ARchive, which is a packaged Java build output.
- You should understand the purpose of `.git`, `.gitignore`, `src`, `build`, `gradle`, and `vendordeps`.
- You should be able to explain the normal uses of `camelCase`, `PascalCase`, `UPPER_SNAKE_CASE`, and `snake_case`.
- You should know how to begin reading a Java project from the outside in.
